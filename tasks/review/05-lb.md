## H-2: EWMA/LeastConn 全局 DashMap 条目永不清理 [高]

### EWMA 指标 DashMap

**文件**: `ewma/metrics.rs` 第 22 行

**问题描述**:

```rust
static EWMA_VALUES: LazyLock<DashMap<SocketAddr, AtomicU64>> = LazyLock::new(DashMap::new);
```

`EWMA_VALUES` 是一个全局 `DashMap<SocketAddr, AtomicU64>`，用于存储每个后端地址的 EWMA 延迟统计。虽然提供了 `remove()` 函数（第 74 行），但**需要外部调用者主动调用**。

**泄漏场景**:

在 Kubernetes 环境中，Pod IP 频繁变化（如滚动更新、HPA 扩缩容）。每次新 Pod 出现时，其新 IP 被添加到 EWMA_VALUES。旧 Pod 被删除后：
- EndpointSlice 数据层会正确移除旧 Pod
- 但 `EWMA_VALUES` 中的旧 `SocketAddr` 条目不会被自动清理
- 长时间运行后，EWMA_VALUES 会积累大量过期的 SocketAddr → AtomicU64 条目

每个条目占用约 40-80 字节（SocketAddr + AtomicU64 + DashMap 开销），在频繁滚动更新的集群中，可能在数周内积累数万个废弃条目。

### LeastConn 连接计数器 DashMap

**文件**: `leastconn/counter.rs` 第 11 行

**问题描述**:

```rust
static CONNECTION_COUNTS: LazyLock<DashMap<SocketAddr, AtomicUsize>> = LazyLock::new(DashMap::new);
```

与 EWMA 相同的问题。虽然 `BackendCleaner`（`leastconn/cleaner.rs`）会清理 draining 状态的后端，但只清理 `backend_state` 和 `counter`，**不清理 `EWMA_VALUES`**。且 BackendCleaner 仅在后端进入 draining 状态后才工作，如果后端被直接从 EndpointSlice 移除（而非 graceful drain），counter 和 backend_state 中的条目也不会被清理。

**建议修复方案**:

方案 A（推荐）：在 EndpointSlice `update_lb_if_exists` 中，检测被移除的后端地址，调用 `ewma::remove()` 和 `counter::remove()` 清理：

```rust
fn update_lb_if_exists(&self, service_key: &str) {
    // ... existing code ...
    // 在 LB 更新后，获取当前活跃后端列表，清理不在列表中的计数器
    if let Some(removed_addrs) = lb.get_removed_backends() {
        for addr in removed_addrs {
            ewma::metrics::remove(&addr);
            leastconn::counter::remove(&addr);
        }
    }
}
```

方案 B：定期清理任务。扩展现有的 `BackendCleaner` 任务，定期扫描 `EWMA_VALUES` 和 `CONNECTION_COUNTS`，对比当前活跃的 EndpointSlice 后端列表，清理不在列表中的条目。

## 审查通过的子模块

| 子模块 | 审查结论 |
|--------|---------|
| RoundRobin | 无状态，仅轮询计数器（AtomicUsize），无泄漏风险 |
| WeightedSelector | 权重表在 LB 重建时完整替换，无泄漏 |
| ConsistentHash | 哈希环在 LB `update` 时完整重建，旧环通过 Arc 回收 |
| BackendCleaner | 正确清理 draining 后端的 state + counter |
| LB Policy Store | ArcSwap 使用正确，`cleanup_lb_policies_for_routes` 清理过期策略 |
| EWMA selection | 算法实现正确，无额外状态分配 |
| LeastConn selection | 算法实现正确，通过全局 counter 读取 |
